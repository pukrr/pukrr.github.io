伪加密

zip伪加密

源于一道许心痕师傅的misc题。其中一部分flag在一个压缩包中，挤压需要密码，然而没有任何关于密码的线索，所以尝试扔到ARCHPR中爆破。但是文件打开错误。

原理：伪加密是在文件头的加密标志位做修改，进而被识别成加密压缩包。
注：不展开说明zip文件格式，仅就事论事。
详情请见：
https://blog.csdn.net/a200710716/article/details/51644421
https://www.bugku.com/thread-182-1-1.html
https://pkware.cachefly.net/webdocs/APPNOTE/APPNOTE-6.2.0.txt
通常用到的zip文件格式：

Offset Bytes Description 翻译
0 4 Local file header signature = 0x04034b50 (read as a little-endian number) 文件头标识，值固定(0x04034b50)
4 2 Version needed to extract (minimum) 解压文件所需 pkware最低版本
6 2 General purpose bit flag 通用比特标志位(置比特0位=加密)
8 2 Compression method 压缩方式（详情见后）
10 2 File last modification time 文件最后修改时间
12 2 File last modification date 文件最后修改日期
14 2 CRC-32 CRC-32校验码
18 4 Compressed size 压缩后的大小
22 4 Uncompressed size 未压缩的大小
26 2 File name length (n) 文件名长度
28 2 Extra field length (m) 扩展区长度
30 n File name 文件名
30+n m Extra field 扩展区


这是我对同一个文件进行加密压缩和非加密压缩的对比。可以看见比较确实有差。
对于真正的加密压缩，如下图。在压缩源文件数据区和目录区的加密位都是09 00。

而对于非加密压缩，如下图。压缩源文件数据区和目录区的加密位都是00 00。而要是改成伪加密，需要将数据区的00 00改成01 00，若是将目录区的00 00改动则没有效果。

这里说明一下：在某个文章中看到说是只要的偶数就是加密，例如00、02、04这样的，奇数就是有加密，例如01 03 09。

用010editor打开这个文件。注：一个字节=两个16进制位

发现加密位部分是01，是有加密的，改为00，可以成功打开压缩文件，不需要密码。
